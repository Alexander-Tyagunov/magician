# nextjs — App Router & Server Components

Current stable: Next.js 16.x. App Router (`app/`) stable since **13.4** (introduced 13.0); Pages Router (`pages/`) still fully supported and can coexist in the same app. Verify the project's `next` version and React version before writing — behavior diverges sharply across 14/15/16.

Version anchors that change how you write code:
- **Server Components by default** in `app/` — since 13.0.
- **Async request APIs** (`params`, `searchParams`, `cookies()`, `headers()`, `draftMode()`) — became **Promises in Next.js 15** (breaking). Sync access still works with a dev warning via casts, but write async.
- **React 19 is the minimum** for Next.js 15+. React 18 only on Next 14 and earlier.
- **`fetch` is NOT cached by default** since Next.js 15 (was `force-cache` in 14). `GET` Route Handlers also uncached by default since 15.
- **Server Actions** stable since Next.js 14.

Detect the router first: `app/` → App Router; only `pages/` → Pages Router. Never mix conventions within one router.

## Server vs Client Components

DO:
- Keep components as Server Components (the default). Fetch data, read secrets/env, hit DBs directly in them.
- Add `'use client'` **only** at the leaf that needs interactivity (state, effects, event handlers, browser APIs, `useContext`). Push the boundary as deep as possible.
- Pass data Server→Client via **serializable props** only.
- Interleave: pass a Server Component as `children`/prop into a Client Component (e.g. `<Modal>{<Cart/>}</Modal>`) to keep server UI inside a client shell.
- Mark server-only modules with `import 'server-only'` to fail the build if imported client-side.

DON'T:
- Don't add `'use client'` to a file just because a child needs it — the directive is transitive: every import in a `'use client'` module graph ships to the browser.
- Don't `useState`/`useEffect`/`onClick`/`window`/`localStorage` in a Server Component — runtime error.
- Don't put React Context providers in Server Components; Context is unsupported there. Wrap `children` in a `'use client'` provider and render it from a layout.
- Don't pass functions, class instances, Dates-with-methods, or other non-serializable values as props to Client Components.
- Don't leak secrets: only `NEXT_PUBLIC_`-prefixed env vars reach the client; others become `""` in client bundles.

```tsx
// app/[id]/page.tsx — Server Component (default), async allowed
export default async function Page({ params }: { params: Promise<{ id: string }> }) {
  const { id } = await params            // Next 15+: params is a Promise
  const post = await getPost(id)         // fetch runs on server
  return <LikeButton likes={post.likes} /> // serializable props only
}
// app/ui/like-button.tsx — Client Component
'use client'
import { useState } from 'react'
export default function LikeButton({ likes }: { likes: number }) {
  const [n, setN] = useState(likes)
  return <button onClick={() => setN(n + 1)}>{n}</button>
}
```

## Routing, layouts, templates

File conventions in `app/` (`.js|.jsx|.tsx`): `page` (route UI, makes segment public), `layout` (shared shell, preserves state across navigation), `template` (re-mounts per navigation), `loading` (Suspense fallback), `error` (error boundary, must be `'use client'`), `not-found`, `route` (Route Handler / API), `default` (parallel-route fallback).

DO:
- Provide the **required root layout** `app/layout.tsx` with `<html>` and `<body>` — Next does not inject them. It replaces `_app`/`_document`.
- Use folders for segments, `[slug]` for dynamic segments, `[...slug]` catch-all, `(group)` route groups (no URL segment), `@slot` parallel routes.
- Use `layout` for persistent chrome (nav/sidebar) — it does **not** re-render on navigation.
- Use `template` instead of `layout` when you need fresh state/effects on every navigation (e.g. enter animations, per-view logging).
- `await params` / `await searchParams` (Next 15+); in a sync Client Component unwrap with `use(props.params)`.
- Prefer the generated `PageProps<'/blog/[slug]'>` / `LayoutProps<'/dashboard'>` global helpers for typing (generated by `next dev`/`build`/`typegen`).

DON'T:
- Don't use `pages/`-style `getServerSideProps`/`getStaticProps`/`getInitialProps` in `app/` — replaced by async components + `fetch` options. `getStaticPaths` → `generateStaticParams`.
- Don't import `useRouter` from `next/router` in `app/` — use `next/navigation` (`useRouter`, `usePathname`, `useSearchParams`, `useParams`); those hooks are Client-only.
- Don't read `searchParams` in a layout — it's page-only and opts the page into dynamic rendering.

## Streaming & Suspense

DO:
- Drop `loading.tsx` in a segment for an instant fallback — Next auto-wraps `page` + children (not the sibling `layout`/`template`/`error`) in `<Suspense>`.
- Use explicit `<Suspense fallback>` around slow async Server Components for granular streaming; multiple boundaries stream independently.
- Colocate `fetch` inside the component that needs it; let independent fetches suspend separately rather than awaiting all upfront.

DON'T:
- Don't block the whole route on one slow query — isolate it behind its own Suspense boundary.
- Don't expect status codes to change after streaming starts (headers already sent); call `notFound()`/`redirect()` before the first suspending `await`.

## Data fetching & caching (App Router)

DO:
- `fetch(url, { cache: 'force-cache' })` for static (≈`getStaticProps`); `{ cache: 'no-store' }` per-request (≈`getServerSideProps`); `{ next: { revalidate: N } }` for ISR.
- Deduplicate cross-call fetches with React `cache()` (e.g. share a query between `generateMetadata` and the page).
- Use `generateStaticParams` to prerender dynamic segments; control unknown params with `export const dynamicParams` (`true` default = on demand, `false` = 404).
- Read request data via async `cookies()`/`headers()` from `next/headers` (Server Components only).

DON'T:
- Don't assume `fetch` is cached — it isn't since Next 15; opt in explicitly. Don't fetch in `useEffect` when a Server Component `fetch` suffices.

## Metadata API

DO:
- Export a static `metadata: Metadata` object, or an async `generateMetadata()` for data-driven tags — **Server Components only**.
- Use file conventions: `favicon.ico`, `icon.*`, `apple-icon.*`, `opengraph-image.*`/`twitter-image.*`, `robots.txt`, `sitemap.xml`. Generate dynamic OG images with `ImageResponse` from `next/og` (flexbox subset only, no CSS grid).
- Use a separate `viewport` export / `generateViewport` for viewport & theme-color.

DON'T:
- Don't use `next/head` in `app/` — it's Pages Router only; use the Metadata API.
- Don't put `metadata`/`generateMetadata` in a `'use client'` file — unsupported.

## Pages Router fallback (still valid)

Repo uses `pages/`: pages are Client Components; use `getServerSideProps`/`getStaticProps`/`getStaticPaths`, `next/head`, `useRouter` from `next/router`, `pages/api/*` handlers. Don't retrofit App Router conventions there. Cross-router shared components: `next/compat/router`.
